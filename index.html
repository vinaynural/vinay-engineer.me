PROJECT_02 / 2024
SAAS · OMNICHANNEL · AI ↗ LIVE PRODUCT

Commai — Omnichannel
AI Communication SaaS

PROBLEM

Businesses communicate with customers across WhatsApp, Instagram, Facebook Messenger, and Email — each requiring a separate tool, login, and workflow. Support teams context-switch constantly, miss messages across channels, and have no unified view of customer conversations. Manual response handling created slow turnaround times and inconsistent customer experience.

SOLUTION

Built Commai — a multi-tenant SaaS backend that unifies all four messaging channels into a single Django-powered API. Each channel is integrated via its respective webhook gateway: WhatsApp Business API, Instagram Graph API, Facebook Messenger API, and Email APIs all route inbound events through a normalised message processing pipeline. Incoming messages are queued via Redis for async processing, persisted to PostgreSQL with full conversation threading, and surfaced to businesses through a unified dashboard. An AI processing layer analyses incoming messages and generates automated reply suggestions, reducing manual response load and improving first-response times across channels.

ARCHITECTURE
Customer Message → WA / IG / FB / Email → Webhook Gateway → Django API → Redis Queue → PostgreSQL → AI Processing → Response Service

Each tenant (business) is isolated at the database level. Webhook endpoints are channel-specific but converge into a shared message normalisation layer before hitting the queue — this keeps business logic channel-agnostic and makes adding new integrations a matter of writing a new webhook handler, not restructuring the core system.

→ WHY REDIS CACHING

Read-heavy endpoints (user profiles, dashboard aggregates, dropdown data) are expensive to recompute on every request. Redis serves these from memory with sub-millisecond latency. Cache invalidation is scoped to the model's signal handlers — data is always consistent without polling the database on every hit.

→ WHY ASYNC WORKERS (CELERY)

Network I/O — sending emails, calling external APIs, processing uploaded files — should never block the HTTP request cycle. Celery decouples these into background tasks, keeping API response times consistently under 200ms regardless of downstream latency. Failed tasks are retried with exponential backoff without any user-facing impact.

→ WHY NGINX AS REVERSE PROXY

Nginx handles TLS, static file serving, and connection management at the OS level — far more efficiently than Python. Django only sees application-level requests. This also makes horizontal scaling (multiple Gunicorn workers) trivial to configure without changing application code.
